# 遗忘机器 → AI 记忆中间件：架构设计

> 固化到项目内，避免遗忘。完整规划文档。

## 核心命题

本文档中的“AI 记忆层”更准确地说，应理解为 **Agent 基础设施层的可衰减记忆中间件**：

- `SKILL.md` / rules 解决的是策略层
- Agent framework 解决的是编排层
- Echo Fade Memory 解决的是“怎么存、怎么衰减、怎么召回、怎么回源、怎么遗忘”

它不是会话内上下文管理器，而是服务于跨会话、长周期、重要但低频信息的长期记忆执行层。

| 现状            | 目标               |
| ------------- | ---------------- |
| 上下文塞满 or 清空重来 | 连续谱：清晰 ↔ 模糊 ↔ 消散 |
| 无差别存储         | 选择性遗忘：噪音下沉，重要沉淀  |
| 数据堆砌          | 人格形成：关联、强化、轮廓    |

### 人格视角

人类的人格恰恰是在**选择性记忆**中形成的：记住的是价值观、习惯、情感连接，而不是每顿饭吃了什么。AI 若模拟这种机制：

- **衰减曲线**：信息随时间模糊，重要度高的事件衰减慢
- **情感加权**：用户强烈表达过的偏好、纠正、肯定被强化
- **人格浮现**：不是预设人设，而是从长期交互中「长」出来的

| 极端 A               | 极端 B                | 本方案              |
| ------------------ | ------------------- | ---------------- |
| 全记住 → 噪音淹没信号，上下文爆炸 | 无记忆 → 每次都是陌生人，没有「人」 | 选择性保留 → 人格从记忆中涌现 |

## 引擎优先：五层架构

> 先做时间驱动的记忆退化引擎，而非 UI 或哲学。核心是「存储、衰减、检索、展示」这些基础能力。

| 层       | 职责                             |
| ------- | ------------------------------ |
| **写入层** | 原始记忆如何保存（格式、版本、元数据）            |
| **时间层** | 多久衰减一次、何时触发变形                  |
| **变形层** | 从精确到模糊的规则（摘要 → 关键词 → 残影）       |
| **检索层** | 查到的是原文、摘要、残影，还是联想召回            |
| **展示层** | 把「遗忘感」可视化（可选，Phase 1 可 CLI 先行） |

### 文本退化时间线（MVP 建议）

先做**文本**记忆退化，不做图片变糊——后者容易跑偏到视觉效果。用文本验证「遗忘」是不是产品能力。

| 时间      | 形态                 |
| ------- | ------------------ |
| 第 0 天   | 完整原文               |
| 第 7 天   | 压缩成摘要              |
| 第 30 天  | 只保留关键词 + 情绪标签      |
| 第 90 天  | 只剩模糊片段             |
| 第 180 天 | 默认不可直达，只能被**联想召回** |

## 0. 四个深挖方向

### 0.1 遗忘的本质：模糊化而非丢失

人类的遗忘不是「丢失」，而是**抽象化**。你记不住 5 年前某顿晚餐吃了什么，但记得那段时间是否快乐。这种从细节到轮廓的转变，恰恰塑造了性格——留下的是情感基调、价值判断，而非琐碎事实。

**设计启示**：衰减后的 residual 应保留**情感倾向、主题、关系**，而非随机截断。抽象化有方向。

### 0.2 AI 人格困境（已覆盖）

无中间态 → 无人格。本方案即填补此空白。

### 0.3 产品形态：会老去的笔记

可做成「会老去的笔记」：存进去的内容随时间**褪色、模糊、合并**。

- 越久远 → 越抽象（关键词、主题、情感）
- 越近期 → 越清晰（完整文本、细节）
- **记忆衰减曲线本身即人格特征**：不同用户的衰减参数不同，形成的「记忆轮廓」也不同

### 0.4 商业角度：反向卖点

**主打「不被记录的隐私」**：数据会自然消失，不留永久数字痕迹。

- 目标用户：不想留下永久数字足迹的人、隐私敏感者、需要「可遗忘」场景的 B 端
- 差异化：与「永久云存储」相反的价值主张

## 1. 存储模型 (Storage Model)

### 记忆单元 (Memory Unit)

```
Memory {
  id: UUID
  content: Embedding + 原始文本/结构化数据
  created_at: Timestamp
  last_accessed_at: Timestamp
  access_count: int
  importance: float (0-1, 可显式标注或从行为推导)
  emotional_weight: float (0-1)  // 情感加权：用户强烈表达的偏好/纠正/肯定
  linkage: [MemoryRef]  // 与其他记忆的语义/引用关系
  decay_state: {
    clarity: float,      // 当前清晰度 0-1
    residual: "outline"  // 完全衰减后的残留形态
  }
}
```

### 图结构：记忆图 vs 知识图谱

**记忆图 (Memory Graph)**——当前设计：

- 节点 = 记忆单元
- 边 = 引用/共现/语义相似
- 边权重 = 关联强度（随共同激活而增强）

**知识图谱 (Knowledge Graph)**——可选增强：

- 节点 = 实体（人、地、概念、事件）
- 边 = 类型化关系（偏好、认识、发生在、涉及）
- 来源 = 从记忆的 residual 中抽取，或显式标注

| 维度   | 记忆图        | 知识图谱                   |
| ---- | ---------- | ---------------------- |
| 粒度   | 记忆块        | 实体 + 关系                |
| 边类型  | 相似度/引用     | 偏好、认识、发生在…             |
| 用途   | 关联强化、簇分析   | 人格锚点、**三路召回之一**        |
| 何时引入 | Phase 1 必需 | Phase 2 引入，与向量、BM25 并重 |

**实现节奏**：Phase 1 做记忆图 + 向量 + BM25 双路召回。Phase 2 引入知识图谱，作为第三路召回：
- 人格导出：「用户偏好 X」「用户认识 Y」等结构化信息
- 实体级召回：「关于某人/某主题的所有记忆」
- residual 抽象化时抽取实体写入 KG，作为衰减后的「骨架」

## 2. 衰减算法 (Decay Algorithm)

### 多因子衰减模型

**清晰度** `clarity(t)` 由四个因子共同决定：

```
clarity = f(time_decay, access_reinforcement, linkage_strength, emotional_weight)
```

| 因子       | 公式思路                            | 效果            |
| -------- | ------------------------------- | ------------- |
| **时间衰减** | 指数衰减 `e^(-λt)`，远期趋近于轮廓          | 近期细节清晰，远期只剩骨架 |
| **访问强化** | 每次 recall 提升 clarity，类似 Hebbian | 被反复想起的不会忘     |
| **关联强化** | 与高权重记忆相连 → 衰减变慢                 | 形成「记忆簇」，簇心更稳固 |
| **孤立惩罚** | 无入边/出边 → 额外衰减系数                 | 孤立信息自然下沉      |
| **情感加权** | 用户强烈表达的偏好/纠正/肯定 → 衰减系数降低        | 价值观、习惯、情感连接沉淀 |

### 情感加权 (Emotional Weighting)

用户强烈表达过的内容应衰减更慢——这些往往是价值观、偏好、边界，是人格的基石。

**触发信号**（可组合）：
- 显式：「记住这个」「这很重要」「你错了，应该是…」「别再说 X」
- 隐式：情感强度高（愤怒、强烈肯定）、重复提及、强调词
- 行为：同一话题上多次纠正 AI → 该记忆 emotional_weight 提升

**效果**：高 emotional_weight 的记忆在衰减公式中享有「保护系数」，形成长期稳定的「人格锚点」。

### 残留形态 (Residual)：抽象化而非丢失

完全衰减后不是删除，而是**抽象化**——保留轮廓，丢掉细节。符合人类遗忘本质。

**文本退化形态**（与 MVP 时间线对齐）：
- 第 7 天：摘要
- 第 30 天：关键词 + 情绪标签
- 第 90 天：模糊片段
- 第 180 天：默认不可直达，仅能**联想召回**（向量/语义触发）

**原则**：抽象化有方向——留下的是可塑造人格的信息（价值、情感、关系），而非随机截断。图像退化（边缘、色块）留待 Phase 2+。

## 3. 召回机制 (Recall Mechanism)

### 召回路径：三路并重

| 路径           | 适用场景                     | 实现                         |
| ------------ | ------------------------ | -------------------------- |
| **向量检索**     | 语义相似、概念联想、「好像记得」         | LanceDB + nomic-embed-text |
| **BM25 关键词** | 精确词匹配、专有名词、代码/配置         | Bleve 全文索引                 |
| **知识图谱**     | 实体级查询、「关于 X 的所有」「用户偏好 Y」 | 实体 + 关系索引（Phase 2）         |

三者互补，缺一不可：
- **向量**：意思相近但用词不同，模糊联想
- **BM25**：必须包含某词，精确命中
- **知识图谱**：结构化关系，人格锚点、实体聚合

**融合策略**：三路各取 top-k → RRF 融合 → 按 clarity 过滤。KG 在 Phase 2 引入后参与融合。

**联想召回**：clarity 极低（如 180 天）的记忆，默认不可直接 key 查、不可 BM25 精确命中，只能通过**向量/语义联想**被触发——模拟「好像记得」「有个模糊印象」。

### 召回流程（三路融合）

```
recall(query) =
  merge(
    vector_search(query, lance_db),    // 语义召回
    bm25_search(query, bleve_idx),     // 关键词召回
    kg_search(query, entity_graph)     // 知识图谱召回（Phase 2 引入，Phase 1 可返回空）
  ).rrf_fusion()
  .filter(m => m.clarity > recall_threshold)
  .rank_by(relevance × clarity)
  .annotate_with(confidence = f(clarity, relevance))
```

## 4. 实现路径建议

### 产品形态选项

| 形态     | 描述              | 用户价值            |
| ------ | --------------- | --------------- |
| AI 记忆层 | 与 LLM 集成的智能记忆后端 | 人格涌现、长期对话       |
| 会老去的笔记 | 独立笔记应用，内容随时间褪色  | 记忆衰减曲线即人格、反完美记忆 |
| 隐私优先存储 | 数据自然消失的存储服务     | 不被记录的隐私、无永久数字痕迹 |

可先做核心引擎，再按场景拆分为不同产品形态。

### Phase 1: 核心引擎（Go）

**目标**：能日用的 CLI，先验证「遗忘」是产品能力而非艺术效果。

- **Go** 实现，单二进制部署，**CLI 先行**（展示层可后续加 UI）
- **部署友好**：`DATA_PATH` 可配置，存储集中；Dockerfile + docker-compose 从 Phase 1 起提供
- **跨平台**：Go 交叉编译支持 macOS (Intel/Apple Silicon)、Windows、Linux；GitHub Releases 多架构二进制；Homebrew/Scoop 等包管理可后续接入
- 五层落地：写入层（存储格式）→ 时间层（衰减调度）→ 变形层（摘要/关键词/残影规则）→ 检索层（三路召回）→ 展示层（CLI 输出）
- **MVP 仅文本**：不做图片变糊，先做文本退化（0/7/30/90/180 天时间线）
- 记忆单元 + **LanceDB** + **Bleve** + Ollama 嵌入
- 衰减算法：对齐上述时间线，可配置 λ、强化系数
- 召回：向量 + BM25；clarity 极低时（如 180 天）仅支持**联想召回**，不可直接 key 查

### Phase 2: 知识图谱 + 与 LLM 集成

- **知识图谱**：实体抽取、关系建模，作为第三路召回参与 RRF 融合
- **HTTP API**：`serve` 模式暴露 REST，供 skill/agent 远程调用
- **Skill 模板**：提供 `echo-fade-memory` skill 示例，描述触发场景与调用方式
- **OpenClaw 插件**：TypeScript 包，注册 `store_memory`/`recall_memory` 等 Agent Tools，通过 HTTP 调用核心服务；`openclaw plugins install` 安装
- 记忆层作为 RAG 的「长期记忆」后端
- 上下文窗口 = 近期清晰记忆 + 模糊轮廓（按需展开）
- 对话中引用记忆时自动触发强化

### Phase 3: 人格涌现

- 分析记忆图：哪些簇稳定（高 emotional_weight + 高 linkage）、哪些已消散
- 导出「记忆轮廓」作为 personality prompt——**不是预设人设，而是从长期交互中长出来的**
- 实验：同一 base model + 不同记忆历史 → 不同「人格」
- 情感加权 pipeline：检测用户强烈表达 → 提升对应记忆的 emotional_weight
- 知识图谱支撑人格描述：「用户偏好 X」「认识 Y」等结构化信息

## 5. 技术选型

### 开发语言：Go

| 考量   | 说明                                                                           |
| ---- | ---------------------------------------------------------------------------- |
| 性能   | 存储密集型应用表现好，GC 对此类场景友好                                                        |
| 开发效率 | 相比 Rust/C++ 迭代快，开源项目需快速验证                                                    |
| 部署   | 单二进制，无运行时依赖，用户拿来即跑                                                           |
| 生态   | LanceDB 有 [lancedb-go](https://pkg.go.dev/github.com/lancedb/lancedb-go) SDK |
| 场景匹配 | 存储 + 时间衰减 + 向量检索，Go 足够                                                       |

> 若追求极致性能的底层引擎可考虑 Rust；做产品、做开源，Go 性价比更高。

### 已选定

| 组件      | 选型                            | 配置                   |
| ------- | ----------------------------- | -------------------- |
| 向量存储    | **LanceDB**                   | 本地、轻量、无服务            |
| BM25/全文 | **Bleve**                     | BM25 评分、RRF 融合、多语言分词 |
| 嵌入      | **Ollama + nomic-embed-text** | 本地推理，768 维           |

```json
{
  "apiKey": "ollama",
  "model": "nomic-embed-text",
  "baseUrl": "http://host.docker.internal:11434/v1",
  "dimensions": 768
}
```

> `host.docker.internal`：应用在 Docker 内运行时，通过此地址访问宿主机上的 Ollama。

## 6. 部署与集成

### 部署形态

| 形态         | 说明                                         |
| ---------- | ------------------------------------------ |
| **直接运行**   | 单二进制，`./echo-fade-memory serve`，无运行时依赖     |
| **Docker** | Dockerfile + docker-compose，数据卷挂载          |
| **存储可移植**  | 所有数据集中在 `$DATA_PATH`（默认 `./data`），可整体备份/迁移 |

**存储目录结构**（便于备份）：

```
data/
├── lancedb/      # 向量
├── bleve/        # 全文索引
├── memories.db   # SQLite 记忆元数据
└── config.json   # 可选覆盖
```

备份：`tar -czvf backup.tar.gz data/`；迁移：解压到新环境即可。

### Skill 触发友好

| 方式             | 适用场景                                                              |
| -------------- | ----------------------------------------------------------------- |
| **CLI**        | Skill 内 `exec` 调用：`echo-fade-memory store "..."`、`recall "query"` |
| **HTTP API**   | REST：`POST /memories`、`GET /memories?q=...`，skill 或 agent 远程调用    |
| **MCP Server** | 可选，作为 Cursor/IDE 的 MCP 工具，AI 可直接 invoke                           |
| **Skill 模板**   | 提供 `echo-fade-memory.skill` 示例，描述何时触发、如何调用                        |

**设计原则**：接口简单、无状态、可配置。Docker 通过 env 指定 `OLLAMA_URL`、`DATA_PATH`、`PORT`。

### 跨平台客户端安装

| 平台          | 安装方式                                               |
| ----------- | -------------------------------------------------- |
| **macOS**   | Homebrew (`brew install echo-fade-memory`)、直接下载二进制 |
| **Windows** | Scoop、Chocolatey、winget、直接下载 `.exe`                |
| **Linux**   | 直接下载二进制、各发行版包管理器（后续）                               |

**Go 交叉编译**：单代码库，`GOOS=darwin/windows GOARCH=amd64/arm64` 产出多平台二进制。Apple Silicon (arm64) 与 Intel (amd64) 均支持。

**发布**：GitHub Releases 附带 `echo-fade-memory-darwin-arm64`、`echo-fade-memory-darwin-amd64`、`echo-fade-memory-windows-amd64.exe` 等；Homebrew tap、Scoop bucket 可后续补充。

### OpenClaw 插件

作为 **OpenClaw** 的插件，将记忆能力暴露给 AI Agent：

| 组件     | 说明                                                       |
| ------ | -------------------------------------------------------- |
| **核心** | Go 服务，HTTP API，独立运行                                      |
| **插件** | TypeScript/JS 包，`api.registerTool()` 注册 Agent Tools      |
| **工具** | `store_memory`、`recall_memory`、`list_memories` 等，LLM 可调用 |
| **安装** | `openclaw plugins install @scope/echo-fade-memory`       |
| **配置** | 插件内通过 env/config 指定 `ECHO_FADE_MEMORY_URL`（核心 API 地址）    |

插件作为 opt-in 工具（`optional: true`），用户需在 `agents.list[].tools.allow` 中显式启用。核心服务可本地运行或 Docker 部署，插件通过 HTTP 调用。
